# 知识库文档摄取 MVP

## Goal

在当前 `feature/chunking-module` 分支上完成一个可验证的知识库文档摄取闭环：文件上传后能够被解析、切分、生成向量，并将分块数据与向量可靠地写入 PostgreSQL/pgvector。

MVP 的目标是先让单个文档的核心链路可运行、可失败恢复、可测试，不扩展为通用摄取平台。

## Background

- 当前分支已经具备知识库、文档、分块和向量的控制器、服务、数据结构及数据库脚本。
- 已存在对象存储、Tika/Markdown 文档解析、分块策略和嵌入模型封装。
- 当前 `mvn test` 会在 Spring 容器启动阶段失败，直接原因是向量实体的 `embedding` 属性没有可用的 MyBatis TypeHandler。
- `C:\etc\ragent` 的对应实现将向量存储作为独立边界，并通过 PostgreSQL 原生 `vector` 转换写入，作为本任务的参考实现。
- 当前数据库脚本已经包含知识库、文档、分块和向量四张核心表；`ragent` 在此基础上额外提供分块执行记录表，以及不属于本 MVP 的定时刷新相关表。
- 当前嵌入模型配置的向量维度为 1536；数据库向量列和索引必须与该维度匹配。

## Requirements

- 仅支持文件上传；URL 抓取、定时刷新不属于 MVP。
- MVP 默认允许上传 `pdf`、`doc`、`docx`、`txt`、`md`、`markdown`、`html`、`htm`，默认单文件上限为 50MB。
- 允许的文件扩展名和大小上限必须配置化，并支持通过环境变量覆盖；后续调整不得修改业务代码。
- 配置修改在应用重启后生效；MVP 不提供后台动态修改或运行时热更新。
- 同一知识库内不允许存在内容完全相同的有效文档；上传前按文件内容计算哈希并进行重复校验。
- 文件类型、文件大小和内容重复校验必须在写入对象存储前完成；对象存储成功但文档记录创建失败时，必须补偿删除本次上传的对象，避免孤立文件。
- 不同知识库允许保存相同内容；同名但内容不同的文件允许上传。
- 仅支持 PostgreSQL/pgvector；Milvus 及多向量后端切换不属于 MVP。
- 上传与文档处理采用两个显式接口：上传接口只保存文件和文档记录，分块接口负责解析、切分、嵌入和持久化。
- 上传成功不得隐式启动完整处理链路；用户可以单独触发首次处理或重新处理。
- 显式分块接口在 MVP 中同步执行；接口调用在处理成功或失败后返回，不创建应用内异步任务。
- 同一文档同一时刻只允许一个处理请求进入；后到请求必须得到明确的“正在处理”错误，状态切换需要使用数据库条件更新避免并发竞态。
- 复用现有对象存储、解析器、分块策略和嵌入模型能力。
- 文档处理必须覆盖解析、分块、嵌入、分块落库、向量落库和文档状态更新。
- 同一文档重新处理时，不得残留上一轮的旧分块或旧向量。
- 重新处理的解析、分块或嵌入阶段失败时，保留上一轮成功的分块和向量；仅当新结果完整生成后才替换旧数据。
- 解析结果为空、未产生任何分块、嵌入为空或嵌入维度不正确都视为处理失败，不得用空结果覆盖上一轮成功数据。
- 新一轮持久化成功时，旧分块、旧向量和文档状态必须在同一数据库事务中完成替换与更新。
- 处理失败时必须标记失败状态并保留可定位的错误信息，不得报告成功。
- MVP 不新增文档分块执行记录表；现有文档记录只保存当前状态和最近一次失败信息。
- 最近一次失败信息保存到文档元数据中；后续处理成功时必须清除旧失败信息，避免状态与错误信息矛盾。
- 写入或清除最近错误时必须合并原有文档元数据，不得覆盖上传阶段已经保存的对象存储等信息。
- 修复当前应用上下文因 pgvector 映射无法启动的问题。
- 普通知识库、文档、分块和状态数据继续使用 MyBatis-Plus；仅向量存储参考 `ragent`，在现有 `KnowledgeVectorService` 边界内改用 `JdbcTemplate` 直接执行 PostgreSQL SQL。
- 向量批量写入必须使用 PostgreSQL 原生类型转换：元数据转为 `jsonb`，嵌入结果转为 `vector`；MVP 不为 `PGvector` 实体字段新增 MyBatis TypeHandler，也不依赖向量实体的 `BaseMapper` 通用 CRUD。
- 数据表设计可以参考 `ragent`，但必须沿用本项目的 `rag` schema、现有命名和已有业务字段，只引入 MVP 实际需要的表或列。
- 数据库必须通过同一知识库下“未删除文档”的内容哈希唯一约束防止并发重复上传，服务层校验用于提供更友好的错误信息。
- 遵守项目模块边界：业务概念留在 `bootstrap`，通用基础设施留在 `framework`。
- 使用构造器注入和 Lombok `@RequiredArgsConstructor`，不使用 `@Autowired` 字段注入。
- 知识库模块的 `*ServiceImpl` 必须补充通俗、能够说明业务目的和关键取舍的注释；服务接口及实现方法使用 Javadoc 描述参数、返回值、失败行为和重要事务边界。

## Acceptance Criteria

- [ ] 上传合法文件后，文档记录和对象存储文件均创建成功，并处于可处理状态，但尚未生成分块和向量。
- [ ] 不在允许列表中的文件会在写入对象存储前被拒绝，并返回明确错误。
- [ ] 超过配置大小上限的文件会被拒绝；修改配置并重启应用后，新限制生效。
- [ ] 同一知识库上传重复内容时，在写入对象存储和创建新文档记录前返回明确的重复错误。
- [ ] 两个并发上传请求包含相同知识库和相同内容时，最多只有一个文档创建成功；失败请求不会遗留对象存储孤儿文件。
- [ ] 不同知识库上传相同内容，以及同一知识库上传同名但内容不同的文件，均不会被误判为重复。
- [ ] 显式调用分块接口后才开始文档处理；重复调用该接口能够发起重新处理。
- [ ] 同一文档正在处理时再次调用分块接口会被拒绝，并且不会启动第二条处理链路。
- [ ] 分块接口同步完成状态迁移、解析、切分、嵌入和持久化；接口返回时文档已处于成功或失败状态。
- [ ] 触发处理后，系统能够读取文件、提取文本、按指定策略切分并生成非空向量。
- [ ] 空文本、空分块、空向量或非 1536 维向量会使本次处理失败，并保留上一轮成功数据。
- [ ] 处理成功后，分块表与 pgvector 表中的数据数量、文档标识和分块标识一致。
- [ ] 文档状态更新为成功，分块数量与实际持久化数量一致。
- [ ] 对同一文档再次处理会替换旧分块和旧向量，不产生重复或孤立数据。
- [ ] 重新处理在持久化前失败时，上一轮成功的分块和向量保持不变，同时文档记录本次失败状态和错误信息。
- [ ] 重新处理成功时，上一轮数据被完整替换，不出现新旧版本混合。
- [ ] 任一处理阶段失败时，文档状态更新为失败，并且不会留下被系统视为成功的部分结果。
- [ ] 状态查询接口能够返回当前处理状态；失败时同时返回最近一次错误信息。
- [ ] 文档重新处理成功后，先前保存的错误信息被清除。
- [ ] 失败信息的写入和清除不会丢失文档原有元数据。
- [ ] Spring 应用上下文能够正常创建向量存储相关 Bean，不再出现 `No typehandler found for property embedding`。
- [ ] 一次文档处理产生的多个向量通过 `JdbcTemplate` 批量写入，而不是逐条循环提交；普通业务表原有 MyBatis-Plus 数据访问方式保持不变。
- [ ] 数据库脚本能够创建与 1536 维嵌入匹配的 pgvector 列和可用的向量索引。
- [ ] 为核心成功路径、失败路径和 pgvector 写入提供自动化测试。
- [ ] 根项目 Maven 测试通过，并能够完成 Maven 打包。
- [ ] 知识库服务接口和 `*ServiceImpl` 的公开方法均有 Javadoc，复杂私有流程有大白话注释且与实际行为一致。

## Out of Scope

- RocketMQ 或其他消息队列。
- Milvus、多向量数据库动态切换。
- 可视化摄取 Pipeline、节点编排和自定义处理流程。
- URL 文档抓取、定时刷新和远程文件变更检测。
- 检索、重排、问答生成和前端管理页面扩展。
- 大规模并发、分布式任务调度和性能压测。
- 历史分块执行记录、各阶段耗时统计和执行日志查询接口。
- MQ、应用内异步线程池和异步任务恢复机制。
- 上传策略的后台管理页面、动态热更新和配置中心集成。
- 同一知识库内故意保存多份完全相同内容的文档。
- 进程被强制终止后自动识别并恢复长期停留在 `processing` 的文档；MVP 仅保证正常异常路径能够标记失败并重试。
